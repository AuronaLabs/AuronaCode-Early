# Aurona Code 0.3.0 技术准备说明

本文档记录 0.2.10 收官版本已经验证的架构边界。它不是 0.2.10 的重构清单；以下项目保持当前实现，留待 0.3.0 在有完整迁移方案与回归验证时处理。

## 已建立的 0.2.10 基线

- 内部性能测试页在应用数据目录保存本地 JSON 基线，并使用真实 Tauri IPC、Rust Rope、搜索和受限临时文件系统路径。
- 启动指标区分前端资源初始化、主界面完成加载和后端到主界面时间；Splash 的两秒展示策略不参与性能结论。
- 工作区搜索采用稳定排序与 500 条上限，前端只接受最新请求结果。
- Linux Release 任务明确安装并验证 GTK/WebKit 依赖，避免把系统依赖缺失误判为 Rust 编译失败。

## 推荐在 0.3.0 处理的边界

1. **编辑器内核拆分**：`AuronaEngine.tsx` 仍承载输入、选择、视口、语法高亮和 LSP 协调等多种职责。0.2.10 只维持其已验证的热路径；0.3.0 应先定义编辑状态与渲染状态边界，再渐进抽取。
2. **IPC 收敛**：`Foundation/IPC` 已提供通用通道，但部分功能仍直接调用 Tauri `invoke` 或插件 API。0.3.0 应建立统一的可测试命令客户端，不能在未完成迁移前同时保留两套业务实现。
3. **状态与事件治理**：Zustand Store、持久化模块和全局 EventBus 目前可用但存在模块级订阅与隐式耦合。0.3.0 应定义生命周期、事件命名与所有权规则，并为高频状态建立专用通道。
4. **插件能力**：当前插件面板明确处于未对外开放状态，没有可交付的扩展 API 或隔离模型。0.3.0 应从权限、生命周期、版本兼容和错误边界设计开始，而不是将占位界面直接扩展为市场入口。
5. **设计系统事实校准**：现有 `docs/Architecture.md` 与 `docs/FileTree.md` 含有部分目标态表述。0.3.0 应以代码为准重写架构文档，并把产品愿景与已实现能力分开说明。
